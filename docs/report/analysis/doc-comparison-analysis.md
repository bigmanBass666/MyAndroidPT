## 三、核心发现

### Set A 的增强内容（B 没有的）

1. **代码示例**：每一实训都提供了可直接运行的 Java 代码，B 只有图片引用
2. **错误纠正**：`Ativity` → `Activity`，`sharedPreferenced` → `SharedPreferences`
3. **MySQL 8.0 适配**：A 区分了 MySQL 5.x 和 8.0 的授权命令，B 只用了旧语法
4. **教学结构**：小结表、作业要求清单、衔接说明（实训间的递进关系）
5. **自查清单**：功能完整性、代码规范、界面设计三重 checklist
6. **项目特殊说明**：SQLite 替代 MySQL 的决策记录
7. **合并策略**：阶段二 README 清晰记录了 5 处源文档冲突及其决策

### Set B 的保留价值（A 没有的）

1. **图片引用**：B 保留了 `![截图](media/image*.png)` 引用，A 完全无图片
2. **原始排版**：B 保留了"密封线""装订线"等试卷格式标记、签名栏等
3. **原始措辞**：`todo.md` 的"邮箱提交"是原始措辞的忠实记录
4. **课堂派公告原文**：`question.md` 是独立的课堂派公告文本

## 四、合并建议

### 原则

- Set A 是**教学交付物**，应作为主文档维护
- Set B 是**原始素材存档**，不应改动，用于溯源和验证

### 具体方案

#### 方案 A：保留现状（推荐）

保持两套文档并存，不作合并：

```
materials/
├── 1_practical_trainning/
│   ├── *.md                    ← Set A：教学主文档（现有，不动）
│   ├── extracted_md/           ← Set B：原始存档（新提取，不动）
│   └── original_docx/          ← 原始 Office 源文件
│
├── 2_design/
│   ├── *.md                    ← Set A：教学主文档（现有，不动）
│   ├── extracted_md/           ← Set B：原始存档（新提取，不动）
│   └── original_docx/          ← 原始 Office 源文件
```

**优点**：零风险，不破坏现有引用

**缺点**：两套相似文档并存，新手可能困惑

#### 方案 B：索引联通

在 Set A 文档中添加脚注/链接，指向 Set B 的图片资源：

```
在 实训3-Android与MySQL的连接.md 中：
（如需查看原图，参见 extracted_md/实训3_Android与MySQL的连接.md）
```

**适用场景**：读者需要参考原始截图时

#### 方案 C：选择性迁移图片引用

从 Set B 中提取图片引用，补充到 Set A 对应位置。但需要注意：

- Set B 的图片路径 `media/image*.png` 在 `original_docx/` 的 `img/` 子目录中
- 迁移需要处理路径映射，操作成本较高
- 图片画面的正确性需要人工逐张确认

---

## 五、推荐方案

### 立即执行（低风险）

**采用方案 A（保留现状）+ 方案 B（索引联通）**：

1. 保持两套文档并存，**不删除、不合并**
2. 在 Set A 每一实训文档的末尾，添加标准脚注：

   > 📎 原始素材存档：参见 `extracted_md/` 同名文件，含原版截图引用

3. 在 `materials/README.md`（或已有 `index.md`）中说明两套文档的关系：

   > - **根目录 .md**：教学使用的主文档（建议优先阅读）
   > - **extracted_md/**：从原始 docx 提取的存档（含原版截图占位）

### 不推荐

- ❌ 删除 Set A → 丢失大量教学增强内容
- ❌ 删除 Set B → 丢失原始素材存档和图片引用
- ❌ 硬合并两套文档 → 产生庞大文件，处理冲突成本高
