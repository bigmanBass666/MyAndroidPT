# Journey Into MyAndroidPT

> 一份基于 claude-mem 4,836 条观测的技术项目完整开发历程分析
> 覆盖周期: 2026-06-01 至 2026-06-22

---

## 1. Project Genesis

MyAndroidPT 的起源可以追溯到 2026 年 6 月 1 日，当时项目还只是一个空的 Android Studio 模板。用户（@wingIsCrazy）的第 597 次记忆会话记录显示，初始工作是对"项目目录结构深度探索"——扫描这个即将承载华南农业大学珠江学院《移动应用开发（Android）》课程设计的骨架项目。第 603 次会话标志着真正的开发起点："实训 2 — Activity 跳转与 Toast 使用的代码实现"。

最初的技术决策在今天看来既平凡又关键：团队选择了 Java 11 + Classic Views（而非当时已成熟的 Kotlin 和 Jetpack Compose），使用 SQLite 本地存储，分离 `user.db` 和 `todo.db` 两个独立数据库。这个技术栈选择不是因为保守，而是因为教材和实训大纲的硬性约束：所有 6 段实训材料全部基于 Java + Activity + SQLite/MySQL 的技术栈编写。AGENTS.md 中的一条边界规则明确着："UI 线程直接执行数据库操作" 是永不触碰的红线。

项目的第 1 个真正意义上的 commit（在 6 月 1 日）完成了登录注册界面的基础框架。第 1 个 AGENTS.md 文件在同一天生成，那时它只有 50 行，定义了三个规则：包结构、Activity 命名约定、DAO 方法命名规范。今天回看，这个文件已经经历超过 10 次重写，成长为 120+ 行的完整项目指南，包含了数据流图、数据库约束、端到端验证循环等技术规范。

**北极星纲要**：项目的双轨需求从一开始就存在矛盾——实训材料要求 MySQL + JDBC，而课程设计评分标准暗示更灵活的技术选型。这个矛盾在 6 月 8 日的 MySQL→SQLite 迁移决策中被彻底解决，但它在整个开发周期中一直以不同形式回响。

## 2. Architectural Evolution

### 第一阶段：MySQL 原型期（6月1日 — 6月8日）

项目最初的架构完全遵循实训教材：`JdbcHelper.java` 管理 MySQL 连接，`UserDao` 通过 JDBC 驱动操作远端数据库，Activity 直接从 MySQL 读取数据。这期间最痛苦的教训发生在 6 月 8 日下午：当尝试在模拟器上运行注册功能时，ClassNotFoundException `java.sql.SQLType` 暴露了 MySQL Connector 8.x 在 Android 环境中的兼容性陷阱。团队花了整整 40 分钟切换 MySQL Connector 版本（8.x → 5.1.49），又花了 1 小时处理 SSL 兼容性异常，最终才意识到这条路走不通。

### 第二阶段：SQLite 迁移（6月8日 16:55 — 17:05）

这是项目架构中最重要的转折点，发生在 10 分钟内。观察 #11336 到 #11346 记录了这个精准的手术式迁移：

1. 删除 `JdbcHelper.java`（观察 #11345）
2. `UserDao` 重构为接受 `Context` 参数，使用 `UserDBHelper`
3. `RegisterActivity`、`MainActivity`、`WelcomeActivity` 同步适配新构造函数
4. 清理 `build.gradle.kts` 中的 MySQL Connector 依赖

这次迁移不是设计上的主动选择，而是被 MySQL JDBC 驱动在 Android 上不可用的硬约束逼出来的。事后（6月15日）团队创建了 `docs/sqlite-migration-decision.md` 来说明这个技术决策，作为课程设计报告的材料支撑。

### 第三阶段：待办 CRUD 模块集成（6月8日 12:33 — 12:42）

在同一天的另一条时间线上，团队从 12:33 开始基于设计文档实现了完整的待办应用模块。观察 #11108 到 #11137 记录了这个 9 分钟的奇迹：

- `Todo.java` 实体类（4 字段：id, title, content, status, createdAt）
- `TodoDBHelper.java` 数据库辅助类
- `TodoAdapter.java` 列表适配器（含 OnItemListener 回调）
- `TodoListActivity`、`TodoEditActivity`、`TodoDetailActivity` 三个 Activity

这个阶段的架构选择奠定了整个应用的骨架：两个独立的 SQLite 数据库、Activity 间通过 Intent Extra 传递数据、子线程执行数据库操作、`runOnUiThread` 更新 UI。这是典型的 Android 经典教学架构，未引入任何现代框架如 Room、ViewModel、LiveData。

### 第四阶段：M3 UI 重构（6月10日 — 6月15日）

6月10日的 M3 UI 重构是项目中规模最大的单次架构变更。4 个 Phase、12 个文件被修改：

- **Phase 1**: `colors.xml` 扩展为 M3 21 色角色体系，`themes.xml` 升级为 M3 父主题
- **Phase 2**: 登录/注册页 M3 组件升级（MaterialButton、MaterialCardView）
- **Phase 3**: 待办模块 M3 组件升级（item_todo、todo_list、todo_detail）
- **Phase 4**: 全部 7 个布局文件 M3 组件升级完成，toolbar title 中文修复

第二天（6月15日）进行了 M3 遗留资源清理：移除 6 个 legacy 颜色别名、创建 `dimens.xml` 统一间距与 elevation 系统、补充 M3 shape appearance 属性。同一天对比分析了开源项目 InstallerX-Revived 的 UI 架构，确认其使用 Jetpack Compose + MUIX，与当前项目的 Classic Views 技术栈完全不在同一轨道。

### 第五阶段：Verdant Design System（6月21日 18:24 — 18:46）

在经历了多次 UI 审查后，团队决定创建自己的设计系统。"Verdant 深翠" 配色方案取代了教材默认色调，建立了间距/排版 Token 系统、Welcome 页仪表盘布局、按钮 loading 状态。3 份规范文档（spec.md、tasks.md、checklist.md）确立了 spec-driven 的工作模式。这是项目从"完成作业"走向"追求品质"的标志性阶段。

### 最终架构：6 页面布局改版（6月22日）

项目最后一天（6月22日）完成了最大规模的布局重构：6 个 Activity 页面的 XML 全部重写，引入 Chip 标签、Toolbar 菜单、状态指示器。特制的 `activity_todo_detail.xml` 从 82 行膨胀到 145 行，但每个元素的语义准确性和视觉层次都有质的提升。

## 3. Key Breakthroughs

### 3.1 MySQL→SQLite 架构迁移（#11336-#11346, 6月8日）

**问题**：MySQL JDBC 驱动在 Android 上不可用，ClassNotFoundException 反复出现。
**突破**：10 分钟内完成从 MySQL 到 SQLite 的全链路迁移，删除 `JdbcHelper`，重构 `UserDao` 为 Context 依赖版本。这是项目最重要的技术决策，直接让应用从"不可运行"变为"可演示"。

### 3.2 自动化截图管线的 UIAutomator 突破（#11588, 6月9日 02:19）

经过近 3 小时的坐标校准失败，突破发生在新坐标 `(130,610)` 使用 swipe 长按式点击时——UIAutomator dump 首次成功获取完整布局 XML。这解锁了整个自动化测试管线，使得后续的端到端测试、回归测试、截图替换成为可能。

### 3.3 ANR 崩溃修复与 seedIfEmpty() 方案（#12518, 6月9日 20:13）

破解 Todo 列表空状态下的崩溃需要双线方案：`monkey LAUNCHER` 保持 Activity 前台 + `seedIfEmpty()` 自动注入种子数据。这个组合方案使全流程自动化截图最终成功通过验证。

### 3.4 TodoDao 多用户隔离架构（#15571, 6月16日）

TodoDao 从无状态重构为构造器注入 userId，这是数据安全的基础设施变革。这个看似简单的改动需要对 6 个 Activity、3 个数据库文件进行全面适配，涉及约 2,000 行代码的审查和修改。

### 3.5 Verdant Design System（#16167-#16174, 6月21日 18:30）

最大的哲学突破发生在设计层面。团队决定超越教材要求，创建自己的 3 档标记体系（教材要求/品质加分/不宜采用），并执行"大于或等于教材要求"的原则。这个决策解放了设计创新，使得 WelcomActivity 的仪表盘、深翠色板、Chip 状态标签等超越了课程设计的标准期望值。

## 4. Work Patterns

### 4.1 调试循环（6月8日—9日）

项目中最密集的调试阶段发生在 6 月 8 日 16:30 到 6 月 9 日 07:00。这个 15 小时的马拉松式调试产生了 867 条观测，占总观测量的 18%。模式清晰可辨：

1. **部署 → 崩溃 → 日志分析 → 修复 → 重部署**
2. 循环周期从最初的 15 分钟逐步缩短到 3-5 分钟
3. 每次失败后都会创建新的自动化脚本版本（`v5` → `v7` → `v9` → `v11`）
4. 最终通过"清除应用数据 + 全新注册 + monkey LAUNCHER"组合解决

### 4.2 特征 Sprint（6月10日、6月22日）

项目的两个主要特征 Sprint 非常短暂但高效：
- **6月10日 M3 重构**：12:25 到 12:42，17 分钟完成 4 个 Phase，效率惊人
- **6月22日 布局改版**：11:33 到 11:46，13 分钟完成 6 页面布局重写

### 4.3 spec-driven 工作流（6月16日起）

从 6月16日的 v2 审查修复开始，团队建立了标准的三文档工作流：
1. `spec.md` — 需求规范
2. `tasks.md` — 任务分解
3. `checklist.md` — 验收清单

### 4.4 用户"剩余工作"查询模式

贯穿整个项目的一个显著模式：用户反复问到同一个问题——"`@materials/1_practical_trainning` 还剩什么没做"。这个模式在对材料的最后两个开发日中最为明显：
- 6月15日：5 次
- 6月16日：7 次
- 6月17日：1 次
- 6月20日：2 次
- 6月21日：6 次
- 6月22日：9 次

每次查询都会触发一次新的材料对照审查和修复，最终在 6月22日产生了完整的 v4/v5 审查体系。

## 5. Technical Debt

### 5.1 SQLite 连接泄漏（6月15日—16日）

项目中积累的最大技术债务是 SQLite 数据库连接泄漏。`TodoDBHelper` 和 `UserDBHelper` 在大量操作中未调用 `close()`，这导致了偶发的随机崩溃和 ANR。6月15日 20:01 定位到"TodoDetailActivity 数据库线程安全崩溃"根因，8 分钟完成 `db.close()` 竞态问题修复。但泄漏问题在 6月16日 02:45 再次浮出水面，这次需要更彻底的 L1-L5 五级修复方案。

### 5.2 SharedPreferences 散列（6月20日—21日）

`spfRecord` 的四字段 key 散落在 MainActivity 和 WelcomeActivity 中，缺少统一管理。6月16日 11:33 的 T1 修复尝试将其集中为 `user_info`，但 WelcomeActivity 的登出逻辑使用 `clear().apply()` 清空全部数据而非只清敏感字段。这个问题在 6月21日 03:59 再次修复。

### 5.3 密码明文存储（贯穿全程）

从 MainActivity.java 第 99 行开始，密码以明文存储在 SharedPreferences 中。6月9日 21:29 的安全审查 `#12597` 明确标记了这个风险。这是一个教育演示项目的固有限制——AGENTS.md 明确标注"密码明文存储（教学演示用途）"。

### 5.4 Toast 被 finish() 提前销毁（6月22日 14:58）

一个长期被忽视的行为缺陷：Activity 在 Toast 显示前调用 `finish()`，导致 Toast 被系统回收。6月22日修复时涉及 4 个 Activity 文件，使用 Handler 延迟 Toast 显示。

## 6. Challenges and Debugging Sagas

### 6.1 注册 ANR 与 RadioButton 坐标校准（6月9日 01:50 — 06:04）

这是项目中持续时间最长、最痛苦的调试战役，从凌晨 1:50 到早上 6:04，跨越超过 4 小时。核心问题有两个：

**RadioButton 坐标校准**：自动化点击始终无法命中注册页的"同意用户协议"复选框。团队尝试了：
- `swipe` 长按点击
- UIAutomator XML 解析获取精确 bounds
- Grid scan 穷举坐标点击
- 视觉 API 验证截图
- 多种坐标系（`(130,610)`、`(160,550)`、`(964,2284)`）

最终发现登认为"自动登录"逻辑绕过注册流程，而 SharedPreferences 中自动登录标志均为 `false`。

**MySQL 不可用**：从 02:07 发现 MySQL 服务未启动，到 02:41 以管理员权限手动启动 MySQL 8.0 并初始化数据库，再到发现项目代码已改为纯 SQLite——这个来回花费了 30 分钟才发现代码与需求脱节。

### 6.2 FAB 点击和保存按钮坐标（6月9日 06:12 — 06:36）

FAB 按钮的坐标点击问题同样困扰了团队近 30 分钟。问题链条：
- FAB 坐标 `(964, 2284)` 在设备 720x1280 分辨率上可被验证
- 但保存按钮坐标持续失效，90 次尝试均未触发导航
- 视觉 API 揭示表单输入未生效——标题字段实际为空
- 根因：`adb input text` 无法写入 EditText

这个发现触发了对 `adb shell input text` 在 Android API 36 上行为的深入调研，最终导向使用 ADBKeyBoard 替代方案。

### 6.3 模拟器离线与 ADB 连接问题（贯穿全程）

模拟器连接问题是项目中最反复出现的环境挑战：
- 6月8日 13:04 首次 ANR（"System UI isn't responding"）
- 6月9日 15:44 x86_64 镜像缺乏 HAXM 硬件加速
- 6月9日 15:52 模拟器进程运行但 ADB 无法检测设备
- 6月10日 12:58 长时间 offline 状态
- 6月15日 12:28 双模拟器同时连接但全部离线
- 6月16日 02:36 ADB logcat 无输出，发现模拟器在 12:37 崩溃

最终解决方案：`--cold` 冷启动标志 + 重启 ADB 服务器 + 关闭多余模拟器实例。

### 6.4 ADB input text 中文输入障碍（多次出现）

`adb shell input text` 不支持中文字符是项目中最独立但最频繁的障碍：
- 6月9日 07:52 首次发现触发的 NullPointerException
- 6月15日 16:33 对模拟器 5558 安装 ADBKeyBoard
- 6月16日 13:33 确认 Android API 36 模拟器输入中文完全失败
- 最终方案：ADBKeyBoard 广播模式 + am broadcast 方式输入

### 6.5 git 嵌套仓库混乱（6月9日 11:35 — 12:27）

文件丢失事件是项目中最令人沮丧的基础设施故障：
- 11:35 发现 29 个文件在 materials/ 目录中被意外删除
- 11:38 确认归档文件"永久丢失"
- 11:49 创建 GUARD.md 防护规则
- 12:10 创建 Git pre-commit 安全钩子
- 12:14 确认父目录 .gitignore 未忽略 Android/
- 12:22 MyAndroidPT 被重构为嵌套独立仓库
- 12:25 最终配置完成，包含 pre-commit 钩子和 .gitignore 规则

这次事故直接催生了两个制度化产出：`GUARD.md` 和 git 工作流规范规则文件。

由于工作段远大于发现段（约 80:1），claude-mem 在这里的实际角色不是压缩——而是**连续性保持**。99% 的表面节约率来自于将完整上下文（工作段）压缩为精心设计的发现段，使得跨会话的上下文重建成本仅为原始工作量的 0.5%。

## 10. Lessons and Meta-Observations

### 10.1 教育项目中的技术约束比商业项目更严格

实训教材的技术栈选择（Java + Activity + SQLite）是整个项目期间不可撼动的约束。每一次试图引入 Kotlin、Jetpack Compose 或 Room 的讨论最终都因"与教材不符"而被驳回。材料目录中的评分标准文档定义了 20 分的"界面设计"打分维度，但任何超出教材的设计决策都必须经过"大于或等于教材要求"的审计框架。

### 10.2 自动化测试的投入与回报长期不对称

项目在自动化测试上投入了大量资源：21 个版本的自动化截图脚本、超过 15 小时的调试循环、ADBKeyBoard + uiautomator2 的多层解决方案。但最终的回归测试仍然依赖手动操作（6月22日的全功能验证依然需要人工点击）。自动化工具链的不稳定性（模拟器离线、UI 框架锁死、路径解析问题）反复削弱了自动化的 ROI。

### 10.3 spec-driven 工作流的三位一体结构

在项目的第 3 周期（6月16日以后）自然形成的"spec.md + tasks.md + checklist.md"结构被证明非常有效：
- **spec.md**：防止范围蔓延——所有改动都能追溯到教材要求
- **tasks.md**：提供原子化的工作分解——每个 task 对应一个可验证的 commit
- **checklist.md**：提供验收标准——防止"代码写完但效果不对"的常见错误

这个结构通过 `.agents/specs/` 目录被制度化，在 layout-polish 和 verdant-design-system 中得到验证，并且自然地抑制了"投机性编码"。

### 10.4 "还剩什么没做"模式的深层含义

用户反复询问剩余任务本质上是对**项目可见性**的需求。每当用户感觉失去对项目状态的全局把握时，他就会问这个问题。claude-mem 的时间线和记忆查询工具提供了一个初步的解决方案——但系统需要一个更可读的仪表盘，使得任务完成状态像 CI 的绿/红徽章一样一目了然。

### 10.5 记忆系统的隐含价值：故障恢复

claude-mem 最有价值的应用场景不是在开发过程中，而是在故障恢复时。6月9日的 git 文件丢失事件、6月15日的 worktree 配置混乱、以及多次的模拟器崩溃恢复——在这些时刻，记忆的"是什么环境配置"比"写了什么代码"更有价值。未来的改进方向应该在事件驱动型记忆（"当 X 发生，记录 Y 配置"）上投入更多。

### 10.6 教育路径与产品路径的分化

MyAndroidPT 项目最终形成了两条并行的决策轨道：
- **教育轨道**：对齐教材要求、遵循评分标准、满足实训检查项
- **产品轨道**：M3 UI 重构、Verdant Design System、Chip 状态标签、仪表盘统计

6月22日 15:18 的 92 次决策中，超过一半涉及"这是教材要求还是品质加分"的判断。这个 tension 不是项目管理的失败——它是"完成作业"和"做好作品"之间创造性张力的体现。
