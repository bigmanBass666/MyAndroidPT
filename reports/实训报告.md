## 一、实训要求

综合运用 Android UI 设计、Activity 跳转、SharedPreferences 本地存储、SQLite 持久化等知识，实现一个包含登录、注册、记住密码、自动登录四项核心功能的 Android 应用，并通过本地 SQLite 数据库进行用户数据的持久化。

项目 MyAndroidPT 涵盖 6 个连续实训任务：实训 1 创建登录注册页面；实训 2 实现 Activity 跳转与 Toast 校验；实训 3 建立 SQLite 数据库并创建用户表；实训 4 实现注册与登录核心功能；实训 5 通过 SharedPreferences 实现记住密码；实训 6 实现自动登录及注册回填功能。

## 二、实训实现步骤

### 步骤 1：创建 MyAndroidPT 工程、配置资源

使用 Android Studio 创建工程，包名统一为 `com.ljx.pt`。配置 `colors.xml` 定义主题颜色（蓝色为主色调），定义 `styles.xml` 继承 `Theme.AppCompat.Light.DarkActionBar`。

### 步骤 2：编写登录页与注册页布局

`activity_main.xml` 实现登录页：账号/密码 EditText、登录 Button、"记住密码/自动登录" CheckBox、"去注册" TextView。`activity_register.xml` 实现注册页：账号/密码/确认密码三个 EditText、协议同意 RadioButton、注册 Button。

账号标签采用 72dp 固定宽度并右对齐，确保与密码输入框对齐一致。

### 步骤 3：实现 Activity 跳转与 Toast 校验

MainActivity 中通过 `Intent` 跳转 RegisterActivity，使用 `registerForActivityResult` 接收注册成功回传的账号和密码并自动填入输入框。登录按钮点击后进行非空校验，校验失败使用 `Toast` 提示。

**关键跳转代码：**

```java
registerLauncher = registerForActivityResult(
    new ActivityResultContracts.StartActivityForResult(),
    result -> {
        if (result.getResultCode() == RESULT_OK && result.getData() != null) {
            String userName = result.getData().getStringExtra("userName");
            String password = result.getData().getStringExtra("password");
            if (userName != null && password != null) {
                etAccount.setText(userName);
                etPassword.setText(password);
            }
        }
    }
);
```

### 步骤 4：建立 SQLite 数据库与用户表

创建 `user.db` 数据库，表名 `userinfo`，包含 `_id`（自增主键）、`name`（唯一账号）、`psw`（密码）三个字段。SQL 建表语句：

```sql
CREATE TABLE IF NOT EXISTS userinfo (
    _id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    psw TEXT NOT NULL
);
```

### 步骤 5：实现注册与登录功能

`UserDao` 提供 `findByName()` 和 `insert()` 方法。注册页调用 `insert` 写入新用户，成功后通过 `setResult(RESULT_OK)` 回传数据。登录页调用 `findByName` 查询用户并校验密码。

**关键登录代码：**

```java
new Thread(() -> {
    UserDao dao = new UserDao(MainActivity.this);
    User user = dao.findByName(name);
    runOnUiThread(() -> {
        if (user == null) {
            Toast.makeText(this, "用户不存在", Toast.LENGTH_SHORT).show();
        } else if (!user.getPsw().equals(psw)) {
            Toast.makeText(this, "密码错误", Toast.LENGTH_SHORT).show();
        } else {
            // 登录成功
        }
    });
}).start();
```

### 步骤 6：SharedPreferences 实现记住密码

登录成功时调用 `saveLoginState()` 写入 SharedPreferences，存储 `isRemember`、`isAutoLogin`、`userName`、`password` 四个字段。应用启动时通过 `initData()` 读取，若 `isRemember=true` 则自动填入账号密码。

### 步骤 7：实现自动登录与勾选联动

`onCheckedChanged` 实现 CheckBox 联动：勾选"自动登录"时自动勾选"记住密码"；取消"记住密码"时自动取消"自动登录"。若 `isAutoLogin=true` 且账号密码均非空，启动时直接调用 `performLogin()` 进入欢迎页。退出登录时清除 `isAutoLogin` 标志。

## 三、遇到的问题及解决办法

### 问题 1：模拟器 Activity 切换后前台焦点丢失导致截图失效

**原因分析**：通过 `am start` 切换 Activity 后，Android 系统的 Launcher 会立即抢占前台焦点。此时进行截图会捕获到桌面而非目标 Activity，导致多轮截图验证失效。这是 Android 系统行为限制，不是代码问题。

**解决办法**：使用 ADB monkey 命令强制锁定应用前台：

```bash
adb shell monkey -p com.ljx.pt -c android.intent.category.LAUNCHER 1
```

经过测试验证，此命令能在 Activity 切换后稳定保持应用在后台，截图正确捕获目标界面。该方案被固化为自动化验证的标准流程。

### 问题 2：模拟器环境无预置用户数据导致注册流程无法演示

**原因分析**：在干净模拟器环境中，`user.db` 首次启动时没有任何用户数据，无法演示完整的"登录→ToList→注册→回填"交互流程。手动输入账号密码在自动化测试中也难以实现。

**解决办法**：参考 TodoListActivity 的 `seedIfEmpty()` 模式，在数据库首次查询为空时自动插入测试账号（虽然最终选择不修改 userDao 以保持登录模块的纯净性，而是通过手动在模拟器上完成交互来获取截图）。

### 问题 3：主线程数据库操作导致 ANR 风险

**原因分析**：Android 8.0 起严格禁止在主线程执行数据库操作。登录时若直接在当前线程调用 `findByName()`，长期运行会导致 ANR（Application Not Responding）错误。

**解决办法**：所有数据库操作封装在 `new Thread(() -> { ... }).start()` 子线程中，查询结果通过 `runOnUiThread` 回调到主线程更新 UI。此模式贯穿登录、注册全流程。

### 问题 4：注册后登录页账号密码未回填

**原因分析**：若使用 `startActivity` 而非 `startActivityForResult` 启动注册页，注册页的 `setResult` 将无法被子 Activity 接收。

**解决办法**：使用 `registerForActivityResult` API 注册回传回调，注册页成功注册后调用 `setResult(RESULT_OK, intent)` 并 `finish()`，登录页在回调中获取回传的账号密码并自动填入输入框。

## 五、总结

通过 6 个连续实训任务，从零搭建了包含登录注册和待办管理的完整 Android 应用。知识收获方面：掌握了 Android UI 资源体系（布局 XML、颜色/样式配置）、Activity 生命周期与跳转机制、SharedPreferences 本地持久化、SQLite 数据库操作及子线程模型。能力提升方面：从纯 UI 页面逐步叠加业务逻辑和数据层，建立了"界面→逻辑→存储"的完整开发思路。

在实际开发中遇到了 Activity 前台焦点丢失、模拟器数据预置、主线程 ANR、跨 Activity 数据回传等问题，通过查阅文档和反复调试逐步解决，积累了 Android 开发问题排查的实战经验。

不足之处：界面美化程度有限，未采用 Material Design 规范；密码以明文存储存在安全隐患；缺少网络层，数据仅本地存储。改进方向：引入 Room ORM 简化数据库操作、使用 OkHttp 实现远程数据同步、增加密码哈希加密、引入 Material Components 升级界面体验。
